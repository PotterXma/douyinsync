# PRD：单视频上传进度可感知（用户体验）

> **角色**：产品经理（BMAD 规划输入）  
> **状态**：草案（待评审 / 拆 Story）  
> **关联**：Epic 5 可观测性、`ui/dashboard_app.py`、`modules/scheduler.py`、SQLite `videos` 表  
> **日期**：2026-04-24  

---

## 1. 背景与痛点

### 1.1 现状

- 管道在后台静默运行；Dashboard 以 **3s 轮询** 展示**聚合**统计与分账号状态**计数**，不指向「某一条视频」。
- **YouTube 分块上传**可能持续数十秒至数分钟；用户只看到「在跑」或日志里的英文行，**无法回答**：「**现在正在传哪一条？传到多少了？还要多久？**」
- 手动同步与定时任务**互斥**时，托盘/日志提示偏技术，**情绪上像黑盒**。

### 1.2 痛点陈述（用户原话抽象）

| 痛点 | 影响 |
|------|------|
| 不知道当前正在处理哪条视频 | 焦虑、不敢关电脑、重复点手动同步 |
| 不知道上传是进行中还是卡死 | 误报故障、增加客服/自查成本 |
| 失败原因分散在日志里 | 非技术用户无法自助判断 |

## 2. 设计目标与原则

### 2.1 目标（可验收）

1. **可识别**：用户能在 5 秒内说出「当前（或最近）正在处理的一条视频」是谁（标题 / 抖音 ID / 账号标记）。
2. **可阶段**：用户能区分至少 **下载中 → 上传中 → 已完成/失败** 三档（与 DB `status` 对齐）。
3. **可进度（上传）**：大文件上传时，用户能看到 **0–100% 或分块索引/已传字节** 之一（技术实现可二选一，产品侧要求「有数字在动」）。
4. **可恢复**：失败时同一界面能看到 **下一步建议**（重试中 / 已放弃 / 去视频库重置），无需翻日志。

### 2.2 非目标（本期不做）

- 不在抖音侧展示「抓取进度条」（仅聚焦 **已进入管道的单条**）。
- 不做多账号 YouTube 并行队列的复杂甘特图（保留后续 V2）。
- 不替代 Bark 的强提醒；**Dashboard 为默认主界面**，Bark 为补充。

### 2.3 设计原则

- **默认简单**：首屏只突出「当前一条 + 阶段」，详情折叠。
- **与状态机一致**：展示字段必须能映射到 `videos.status`（及未来扩展字段），避免第二套真相。
- **子进程只读**：Dashboard 仍为独立进程时，**只读 DB**（与现有架构一致），进度通过 **DB 或轻量 IPC** 写入（实现阶段由开发定）。

## 3. 用户画像与场景

| 画像 | 典型场景 | 核心诉求 |
|------|-----------|----------|
| 运营个人 | 夜间挂机传长视频 | 「还在传」+ 百分比 |
| 小工作室 | 多账号日限额 1 | 「今天这条算不算成功」 |
| 技术 owner | 排错 | douyin_id、本地路径、最后一次错误摘要 |

## 4. 信息架构：用户可见的「进度」定义

### 4.1 阶段模型（与实现对齐的「用户语言」）

```mermaid
stateDiagram-v2
    direction LR
    pending: 排队待下载
    processing: 准备下载
    downloading: 下载中
    downloaded: 已下待传
    uploading: 上传中
    uploaded: 已上传
    failed: 失败可重试
    give_up: 已放弃
```

**上传进度条**仅在 `uploading` 时有意义；`downloading` 可用下载侧已有日志粒度扩展为「下载 %」（若产品坚持两阶段都有条，开发评估 DB 写入频率）。

### 4.2 「当前活动视频」定义（产品规则）

**优先级**（从高到低，用于 Dashboard 主文案）：

1. `status = 'uploading'` 的**最早 `updated_at`** 一条 → 显示为 **「正在上传」**  
2. 若无，则 `status = 'downloading'` 或 `processing`（若未来细分）→ **「正在下载」**  
3. 若无，则最近 `updated_at` 在 10 分钟内且为 `pending` 且本轮管道 `Lock` 已占用 → **「管道运行中，下一条即将开始」**（弱提示，可选）

## 5. 方案总览（产品形态）

### 5.1 Dashboard（主战场）— MVP 必做

| 模块 | 内容 |
|------|------|
| **当前活动卡片** | 标题截断 + `account_mark` + `douyin_id` 短码；主阶段标签（下载中/上传中）；开始时间（`updated_at` 可读） |
| **上传进度** | 进度条 + 百分比 或「已传 MB / 总 MB」；无活动则显示「空闲」 |
| **最近完成一条** | 成功：绿色勾 + YouTube 视频 id（可点击复制）；失败：红色 + 一行原因（来自 DB 或最后日志摘要字段） |
| **与全局条关系** | 顶部 YouTube 配额条保留；说明文案：「配额条为**当日账号级**；下方为**当前单条**」 |

### 5.2 视频库 / 列表（V1）

- 在 `videolib` 或 Dashboard 内嵌表格增加列：**阶段**、**上传 %**（仅 uploading）、**最近更新时间**。
- 支持按 `status` 筛选（已有能力上增强展示即可）。

### 5.3 托盘与通知（V1–V2）

| 能力 | 说明 |
|------|------|
| 托盘 tooltip / 二级文案 | 单行：`上传中: 标题… 62%`（需主进程写共享状态或轮询 DB） |
| Bark | 可选「仅在上传开始 / 成功 / 失败」推送，避免刷屏（配置项 `bark_upload_events`） |

### 5.4 数据与实现约束（给架构 / 开发）

- **方案 A（推荐）**：扩展 `videos` 表或 JSON 侧车字段：`upload_bytes_done`, `upload_bytes_total`, `last_stage_message`, `last_error_summary`，由 **上传器协程**周期性 `UPDATE`（注意 WAL 与写入频率，建议 1s 或 5% 步进刷盘）。  
- **方案 B**：内存 + 单文件 `pipeline_state.json` 由守护进程写，Dashboard 读文件（实现快，但与「DB 为唯一真相」略偏离，需文档约定）。  

产品侧：**优先 A**，与 BMAD 状态机故事一致。

## 7. 验收标准（UAT）

1. 人为制造一条 `uploading`（或模拟大文件），Dashboard **3s 内**出现该条标题且进度**单调递增**直至完成或失败。  
2. 上传成功：同一界面出现 **YouTube id** 或明确「已成功」且 DB `status=uploaded`。  
3. 上传失败：同一界面出现 **失败原因关键词**（与日志一致级别），且与 `failed`/`give_up` 策略文档一致。  
4. 无上传任务时，**不出现**假进度条乱跳。

## 8. 指标（上线后观察）

- Dashboard 打开率 / 停留时长（若可埋点则埋，否则问卷）。

## 9. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 高频写 DB 影响性能 | 节流更新、批量写、或仅写内存快照文件 |
| 子进程读不到实时字节 | 必须以 DB/文件为媒介，禁止跨进程直接调对象 |
| 错误信息泄露 | 沿用日志脱敏规则，UI 不展示完整 Cookie/Token |
